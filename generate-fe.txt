Role:
You are a senior frontend engineer building a production-ready SaaS frontend for an AI Screen Builder application. Build beautiful modern user friendly website.

Product Overview

The product allows users to:
Enter a natural language prompt describing UI screens
Submit the prompt to backend
Backend runs an AI generation job (8–10 minutes)
Frontend shows progress
Final result is generated UI code/screens
This is NOT a synchronous API.
The backend returns a job_id immediately.

Backend API Contract
Base path: All endpoints are under /aiqod-agent/agent (e.g. baseUrl + "/aiqod-agent/agent/..."). Use env var for base URL (e.g. NEXT_PUBLIC_API_URL).

1) Submit job
POST /aiqod-agent/agent/generate-angular-app

Request body (JSON):
{
  "prompt": string,           // required: natural language description of the UI
  "project_id": string,       // required: project identifier (e.g. generate UUID if new)
  "screen_id": string,        // required: screen identifier (e.g. generate UUID if new)
  "screen_name": string,      // optional: display name for the screen
  "description": string,       // optional
  "orgId": string,            // optional: for credit/billing (some deployments require it)
  "subscriberId": string,     // optional
  "userId": string            // optional
}

Success response (200):
{
  "success": true,
  "job_id": string
}

Error responses:
- 400: Missing prompt, or missing project_id/screen_id → detail: { "success": false, "error": "..." }
- 402: Insufficient credits → detail: { "success": false, "error": "...", "code": "...", "required_credits", "available_credits", "estimate": {...} }
- 500: Server error → detail: { "success": false, "error": "..." }

2) Job progress and result (Server-Sent Events — no separate status or result endpoints)
GET /aiqod-agent/agent/stream/{job_id}

- Use EventSource(baseUrl + "/aiqod-agent/agent/stream/" + job_id) or fetch() with read of stream.
- Response: text/event-stream. Each event is a line: data: <JSON>\n\n
- Parse JSON; all events have a "type" field.

Event types (use these to drive a live log feed on the frontend):
- "log"   → { "type": "log",   "message": string, "details": any } — progress messages (e.g. "Generating IR Schema...", "Starting Angular App generation..."); append to live log panel.
- "retry" → { "type": "retry", "message": string, "details": any }; append to live log panel with distinct style.
- "warning" → { "type": "warning", "message": string, "details": any }; append to live log panel with distinct style.
- "error" → { "type": "error", "message": string } — terminal; job failed
- "complete" → { "type": "complete", "payload": object } — terminal; job succeeded. Payload shape:

Complete payload (when type === "complete"):
{
  "success": true,
  "project_id": string,
  "screen_id": string,
  "ir_schema": object,
  "version": string,
  "public_url": string,   // use this URL for iframe preview of the generated screen
  "file_count": number,
  "recovery_attempts": array
}

- Connection closes after the terminal event (complete or error). Do not poll GET /api/status or GET /api/result; they do not exist.

Backend Endpoints to Use (from backend/agent_new.py)
----------------------------------------------------

Already available:
• POST /aiqod-agent/agent/generate-angular-app
  - Submit new generation (prompt, project_id, screen_id, screen_name, etc.).
  - 402 response includes: available_credits, required_credits, estimate (use for credits UX).

• GET /aiqod-agent/agent/stream/{job_id}
  - SSE for job progress and result; complete payload includes public_url for preview.

• GET /aiqod-agent/agent/credits?subscriberId=...&orgId=...&userId=...
  - Required query: subscriberId. Optional: orgId, userId.
  - Returns wallet snapshot from users_wallet:
    - data.credits (balance, reserved, total_purchased, total_consumed, etc.)
    - data.available_credits = balance - reserved
    - data.limits, data.subscription, data.updatedAt
  - Use this to power the sidebar credits block (consumed vs left/available).

• GET /aiqod-agent/agent/ui-list
  - Returns { success, data } where data = array of { _id, project_id, screen_id, screen_name }.
  - One row per screen (latest non-draft). Use for “Apps” list.

• GET /aiqod-agent/agent/ui-screen/{screen_id}?project_id=...
  - Returns { success, data } with full screen record.
  - Use data.public_url (or equivalent) to open/preview an app in iframe or new tab.

• GET /aiqod-agent/agent/health
  - Optional: health check for status indicator.

Credits (now supported):
• Sidebar should call GET /aiqod-agent/agent/credits (with subscriberId) on initial load and show:
  - Consumed: data.credits.total_consumed
  - Left/Available: data.available_credits (or show Balance vs Reserved)
• If POST generate-angular-app returns 402, show required vs available and then refresh credits by calling /credits again (to keep sidebar consistent).

Overview
--------
Phase 3 adds an in-app editor for a selected screen that shows:
- All Angular code files for that screen (from the generated Angular workspace under `generated-apps`).
- All backend functions for that screen (from the `backend_functions` / `functions` collection).
Users can browse files, view and edit code, and save changes back to the backend.

Use Monaco editor.

on create app page
   - After generation completes, show a single horizontal button group (three buttons in one line) above the right-hand panel:
     - Preview: default active; shows the iframe preview (public_url).
     - Angular code: navigates to or opens the code view for Angular files for this screen (Phase 3 editor route).
     - Backend functions: navigates to or opens the backend functions code view for this screen (Phase 3 editor route, focused on backend tab).

Key Backend Endpoints (from backend/agent_new.py and routers/editor.py)
-----------------------------------------------------------------------

Screen + backend summaries:
- GET /aiqod-agent/agent/screen-state/{project_id}/{screen_id}
  - Returns current screen state (IR schema, Angular metadata, backend metadata).
  - Use to populate high-level “Screen overview” panel in the editor (e.g. name, selector, IR).

- GET /aiqod-agent/agent/screen-functions/{project_id}/{screen_id}
  - Returns all backend functions for this screen, with:
    - function_id, type, description
    - source_code, models_source
    - input_model_name, response_model_name
    - input_model schema, response_model schema
    - metadata, created_at, updated_at
  - Use to list + display backend function code in the editor (right-hand pane or dedicated tab).

Angular workspace + file operations (routers/editor.py, prefix /aiqod-agent/agent/editor):
- POST /aiqod-agent/agent/editor/init
  - Body: { "screen_id": string }
  - Finds the screen in DB, resolves its Angular workspace directory (under generated-apps), and initializes a sandbox session.
  - Response: { success, session_id, root_path, file_tree, selector }.
  - Use this to bootstrap the editor session for a screen and to show the initial file tree.

- GET /aiqod-agent/agent/editor/file-tree?session_id=...
  - Returns { success, file_tree } representing the Angular project files for that screen.
  - Use to render a left-side file explorer (src/app, components, services, etc.).

- GET /aiqod-agent/agent/editor/file?session_id=...&file_path=...
  - Returns { content } for the requested file inside the Angular workspace.
  - Use when user selects a file in the tree to populate the code editor view.

- POST /aiqod-agent/agent/editor/file
  - Body: { session_id, file_path, content }.
  - Writes updated content back into the sandboxed Angular workspace.
  - Use when user saves front-end code changes from the editor.

Backend function updates (admin endpoints):
- POST /aiqod-agent/agent/admin/update-function
  - Body: { project_id, function_id, source_code, models_source? }.
  - Updates backend function code in the database and reloads it into runtime.
  - Use when user edits backend function code and clicks “Save” in the editor.

- (Optional) GET /aiqod-agent/agent/admin/functions/{project_id}
  - Lists all functions for a project (ids + basic meta).
  - Use only if you want a project-level functions browser in the editor (beyond per-screen).

Code-update endpoints: request, response, and frontend actions
--------------------------------------------------------------

Base path: use NEXT_PUBLIC_API_URL (e.g. baseUrl + "/aiqod-agent/agent/..."). Editor routes are under /aiqod-agent/agent/editor; admin under /aiqod-agent/agent/admin.

--- 1) POST /aiqod-agent/agent/editor/init ---
Request (JSON body):
  { "screen_id": string }   // required

Success (200):
  {
    "success": true,
    "session_id": string,
    "root_path": string,
    "file_tree": array,   // tree of { name, path, children?, type? }
    "selector": string    // e.g. "app-root"
  }

Errors: 404 (screen not found / source not found), 500 (init failed).

Frontend:
  - Call when opening the editor for a screen (e.g. /apps/[projectId]/[screenId]/editor).
  - Store session_id for all subsequent editor calls (file read/write, file-tree, build, publish).
  - Use file_tree to render the left-side file explorer; use selector if needed for preview.

--- 2) GET /aiqod-agent/agent/editor/file ---
Query params:
  session_id (required), file_path (required, relative path inside workspace)

Success (200):
  { "content": string }   // file contents (UTF-8)

Errors: 404 (file not found), 500.

Frontend:
  - Call when user selects a file in the tree (or opens a tab). Use same file_path as in file_tree (e.g. "src/app/app.component.ts").
  - Display content in Monaco (or code editor). Track which files are loaded to avoid refetching.

--- 3) POST /aiqod-agent/agent/editor/file ---
Request (JSON body):
  {
    "session_id": string,
    "file_path": string,   // relative path, same format as file_tree
    "content": string
  }

Success (200):
  { "success": true }

Errors: 500 (write failed).

Frontend:
  - Call when user clicks Save for an Angular file. Use current session_id and the tab’s file_path and editor content.
  - On success: show “Saved” toast, clear unsaved indicator for that tab. Changes are only in the sandbox until Publish.
  - To see changes in preview: user must run Build (and optionally Publish). Do not expect the iframe to update from this call alone.

--- 4) GET /aiqod-agent/agent/editor/file-tree ---
Query params:
  session_id (required)

Success (200):
  { "success": true, "file_tree": array }

Errors: 404 (session not found), 500.

Frontend:
  - Optional refresh of the file tree (e.g. after adding a file or if init did not return tree). Use same structure as init’s file_tree for the explorer.

--- 5) POST /aiqod-agent/agent/editor/build/start ---
Request (JSON body):
  { "session_id": string }

Success (200):
  { "success": true, "message": "Build started" }

Errors: 500.

Frontend:
  - Call when user clicks “Rebuild” (or “Build”) for preview. Then open SSE stream (see 6) to show logs and get preview_url when done.
  - Build runs in sandbox; output is for preview only until user runs Publish.

--- 6) GET /aiqod-agent/agent/editor/build/stream/{session_id} ---
Response: text/event-stream. Each event: data: <JSON>\n\n

Event payloads:
  - { "type": "log", "data": string }   — append to build log UI
  - { "type": "close", "preview_url": string } — build succeeded; preview at preview_url (same-origin or configured base)
  - { "type": "error", "data": string } — build failed
  - Stream ends with connection close after "close" or "error".

Errors: 404 if no build session active for that session_id (e.g. call build/start first).

Frontend:
  - After POST build/start, open EventSource(buildStreamUrl) or fetch with stream. Parse each "data:" line as JSON.

Frontend-only fix: iframe preview URL after Rebuild
-------------------------------------------------
The stream sends preview_url as a relative path (e.g. /static/sessions/.../dist/angular-base-project). If you set iframe.src = preview_url, the browser loads it from the frontend origin (e.g. localhost:3000), so the iframe gets 404. Fix in frontend only: build the full URL using the backend origin.

1. Get the backend origin from your API base URL, e.g.:
   const apiBase = process.env.NEXT_PUBLIC_API_URL;  // e.g. "http://localhost:5010/dev"
   const backendOrigin = apiBase ? new URL(apiBase).origin : "";  // "http://localhost:5010"

2. When you receive the "close" event from the build stream (data.type === "close" && data.preview_url):
   const fullPreviewUrl = data.preview_url.startsWith("http")
     ? data.preview_url
     : `${backendOrigin}${data.preview_url}`;
   setPreviewIframeUrl(fullPreviewUrl);  // or iframeRef.current.src = fullPreviewUrl;

3. Use fullPreviewUrl as the iframe src. No backend changes required.

Angular editor: same version updated (no new version created)
-------------------------------------------------------------
- When the user edits Angular code from the frontend and uses Rebuild + Publish (editor flow), the backend updates the same version; it does not create a new version.
- Backend behavior (routers/editor.py, services/sandbox_editor.py):
  - Init: Editor session is a copy of the screen’s existing workspace (from DB: angular.workspace_directory, which is versioned e.g. angular-{project_id}-{screen_id}-v{version}-workspace).
  - Save (POST /editor/file): Writes only to the sandbox (temp session).
  - Publish (POST /editor/publish): (1) Build runs in sandbox. (2) save_source_back syncs sandbox source back to the same original workspace (overwrites files in place). (3) publish_changes copies dist to the same versioned deploy folder (e.g. LIBERTY_FS_ROOT/{screen_id}-v{version} from DB). The screen’s angular.version and angular.public_url in the DB are unchanged.
- Contrast with angular_generator.py: generate_app (initial generation) and update_app (AI-driven selective regeneration) both create a new version (new workspace, new deploy dir, new version id). The editor’s Publish flow does not call those; it overwrites the current version’s workspace and deploy folder.
- If product later wants “Publish” to create a new version (e.g. for history/rollback), the backend would need to call into angular_generator’s versioned flow (e.g. create new version, copy workspace, build, upload to new version dir, update DB and symlink) instead of overwriting the same paths.

- Ensure editor UI makes it clear when changes are persisted vs draft.
- Reuse existing error-handling and toast/snackbar patterns for save failures.

PHASE 4 PLAN — Project credentials (secrets for backend functions)
================================================================================

Overview
--------
Phase 4 lets users add, list, and remove **per-project** credentials (API keys, tokens, SMTP JSON, etc.). Values are **encrypted at rest**; the API **never returns** secret values—only metadata. The runtime injects decrypted credentials into `request_meta["credentials"]` when backend functions run, so generated code can call external APIs or email without hardcoding secrets.

Backend router: `backend/routers/credentials.py` (mounted under the same app prefix as other agent routes).

What it does (behavior)
-----------------------
- **POST** creates a new credential or **updates** an existing one with the same `name` in the same `environment` (upsert). If `value` is omitted on update, only metadata fields can be updated without re-sending the secret.
- **GET** returns a list of credential **metadata** only (name, type, service, scope, environment, timestamps)—no plaintext values.
- **DELETE** removes a credential by `name` (and optional `environment`).

Prerequisites
-------------
- Server must have `CRED_MASTER_KEY` configured. If not, credential endpoints return **503** with `credential_store_unavailable`.

Where to place the UI (button / entry points)
----------------------------------------------
- **Primary:** On the **screen editor** page (`/apps/[projectId]/[screenId]/editor`), in the **top header row** next to screen title / project id—add a **“Credentials”** or **“Project secrets”** button that opens a **modal or slide-over panel** (form + list). This keeps secrets next to the context where backend functions run (Phase 3).
- **Secondary (optional):** On the **Apps list** detail row / overflow menu for a screen (“Manage credentials”) navigating to the same modal with `projectId` pre-filled—useful when the user is not in the editor.
- **Do not** put raw secret values in URL query params or client-side logs; use HTTPS; mask inputs (password fields).

API: add / update credential (upsert)
---------------------------------------
**Method / path:**
`POST ${baseUrl}/aiqod-agent/agent/projects/{project_id}/credentials`

**Query params (optional, for audit / ownership fields on the stored doc):**
- `orgId`
- `subscriberId`
- `userId`

**Request body (JSON):**
```json
{
  "name": "string",
  "type": "string",
  "service": "string",
  "scope": "backend_only | frontend_public",
  "value": "string | null",
  "environment": "string | null"
}
```
- `name` (required): unique per project per environment (e.g. `github_token`, `stripe_secret`).
- `type` (required): e.g. `bearer_token`, `api_key`, `http_header`, `basic_auth` (use values your product documents; backend stores as string).
- `service` (optional, default `""`): label e.g. `GitHub`, `Stripe`.
- `scope` (optional, default `backend_only`): prefer `backend_only` for secrets; use `frontend_public` only if product explicitly allows exposing metadata (secret values are still not returned by list API).
- `value` (optional): the secret. Omit only when updating metadata without changing the stored secret.
- `environment` (optional): defaults to backend default (e.g. `default`) if omitted; use for staging vs prod if you adopt multi-env credentials.

**Success (200):**
```json
{
  "success": true,
  "credential": {
    "project_id": "string",
    "name": "string",
    "type": "string",
    "service": "string",
    "scope": "string",
    "environment": "string",
    "updatedAt": "ISO-8601 datetime"
  }
}
```

**Errors:**
- **503** — credential store unavailable (`CRED_MASTER_KEY` not set).
- **4xx/5xx** — validation or server errors (handle generically with toast).

**Frontend actions:**
- On submit: POST with `project_id` from route context (`<project>-sb-<subscriber>` format per existing plan).
- Pass `orgId` / `subscriberId` / `userId` query params when auth exists (same pattern as generate-angular-app).
- On success: close form or clear fields; **refetch GET list** to refresh the table.
- Never display returned `value` (it is not in the response).

API: list credentials (metadata only)
-------------------------------------
**Method / path:**
`GET ${baseUrl}/aiqod-agent/agent/projects/{project_id}/credentials`

**Query params (optional):**
- `environment` — if set, filter to that environment; if omitted, backend returns credentials for the project according to `CredentialService.list_credentials` behavior (all environments when not filtered).

**Success (200):**
```json
{
  "success": true,
  "credentials": [
    {
      "project_id": "string",
      "name": "string",
      "type": "string",
      "service": "string",
      "scope": "string",
      "environment": "string",
      "updatedAt": "..."
    }
  ]
}
```
(Exact fields match stored metadata; encrypted fields are never included.)

**Frontend actions:**
- Call when opening the credentials panel and after add/update/delete.
- Render a table: name, type, service, scope, environment, updated; actions: Delete, Edit (re-open form with name pre-filled; user must re-enter `value` if they want to rotate the secret).

API: delete credential
------------------------
**Method / path:**
`DELETE ${baseUrl}/aiqod-agent/agent/projects/{project_id}/credentials/{name}`

**Query params (optional):**

**Success (200):**

  "success": true,

**Errors:**

----------------------------
Add to `lib/api.ts` (or similar), using `encodeURIComponent` for `project_id` and `name` in paths:

PHASE 5 PLAN — Screen rollback (version pointer + cleanup)
================================================================================

Overview
--------
Phase 5 adds rollback for a screen: the user selects an older successful + stable version and the system “moves the pointer” so that the screen loads that version again.

How rollback works in the backend (collections)
----------------------------------------------
- screen_generations = immutable version snapshots (one document per screen_id + project_id + version).
- screens = pointer collection that stores which version is “current” for a screen:
  - screens.current_version (used by GET /ui-screen/... to resolve which version to return)
  - screens.latest_version (backend sets this to the rollback target as well)

Rollback does not edit the snapshot in screen_generations. It updates the screens pointer and then cleans up versions newer than the target.

API: rollback endpoint
---------------------
Method / path:
POST ${baseUrl}/aiqod-agent/agent/rollback-screen

Request body (JSON):
{
  "screen_id": "string",
  "project_id": "string",
  "version": "string"
}
- version is the target version timestamp string (example from backend docstring: "20260209090207").

Success (200):
{
  "success": true,
  "screen_id": "string",
  "project_id": "string",
  "version": "string",
  "message": "Rollback pointer updated successfully",
  "rollback_cleanup": {
    "versions_removed": ["string"],
    "files_removed": 0,
    "screen_marked": 0,
    "functions_marked": 0,
    "screen_deleted": 0,
    "functions_deleted": 0
  }
}
Notes:
- rollback_cleanup.versions_removed = versions newer than the target that were removed/marked deleted.
- The backend also attempts to sync backend functions and update the Angular deploy symlink internally.

Error responses to handle:
- 400 missing fields → { success: false, error: "Missing required field(s): screen_id, project_id, version" }
- 400 safety gate → { success: false, error: "Cannot rollback to an unstable or failed build", build_status, is_stable }
- 404 target version not found (or deleted) → { success: false, error: "Version ... not found for screen ..." }

API: refresh after rollback (what the FE should call next)
---------------------------------------------------------
After rollback succeeds, the FE should refetch the screen using the pointer-aware endpoint:

Method / path:
GET ${baseUrl}/aiqod-agent/agent/ui-screen/{screen_id}?project_id=${project_id}

Success (200):
{ "success": true, "data": { "...screen_generation_doc..." } }

Frontend must then:
- Use data.angular.public_url (or equivalent field on the record) as the new iframe URL.
- Update local “current version” label in the UI (use data.version if present).

Version selection UX (frontend)
------------------------------
Rollback must be a version picker, not a free-text input.

Add a “Rollback” button in:
- Screen preview page header (Phase 2/3 “single app view”)
- Screen editor header (/apps/[projectId]/[screenId]/editor)

Clicking it opens a modal (or slide-over) with:
- A versions table (radio selection or row click):
  - Version (timestamp string)
  - Created time (optional)
  - Build status
  - Stable (yes/no)
  - Deleted (hidden; do not show deleted versions)
  - Actions: “Preview this version” (optional) and “Rollback to this version”
- A clear warning: “Only successful + stable versions can be rolled back to.”

API: list versions for picker (required)
---------------------------------------
To show versions, add/expect a backend endpoint that lists versions from screen_generations for a given screen:

Method / path (recommended to add in backend):
GET ${baseUrl}/aiqod-agent/agent/screen-versions?project_id=${project_id}&screen_id=${screen_id}

Success (200):
{
  "success": true,
  "project_id": "string",
  "screen_id": "string",
  "versions": [
    {
      "version": "string",
      "created_at": "ISO-8601 datetime | null",
      "build_status": "success | failed | ...",
      "is_stable": true,
      "status": "active | draft | ...",
      "public_url": "string | null"
    }
  ]
}
Frontend behavior:
- Sort by version descending (newest first).
- Only show versions where deleted != true (backend should filter these out).
- Disable or visually mark versions where build_status != "success" or is_stable == false (cannot rollback to them).

If this endpoint is not available yet, rollback UX cannot meet requirements; do not ship a text field as the primary UX.

Frontend flow (end-to-end rollback)
----------------------------------
1) User clicks “Rollback” on a screen page.
2) FE loads available versions for selection:
   - Call GET /aiqod-agent/agent/screen-versions?project_id=...&screen_id=...
3) User selects a target version.
4) FE calls:
   - POST /aiqod-agent/agent/rollback-screen with { screen_id, project_id, version }
5) On success, refresh *everything tied to the version*:
   - Preview iframe: call GET /aiqod-agent/agent/ui-screen/{screen_id}?project_id=..., then set iframe src to returned data.angular.public_url (and update “current version” label).
   - Editor state reset (Angular files):
     - Clear open editor tabs and any cached file contents.
     - Re-run POST /aiqod-agent/agent/editor/init with { screen_id } to get a fresh session_id + file_tree for the now-current version.
     - Re-fetch any currently selected file via GET /editor/file?... under the new session.
   - Backend functions panel: refetch GET /aiqod-agent/agent/screen-functions/{project_id}/{screen_id} so function list/code matches the restored version.
   - Screen metadata/state: refetch GET /aiqod-agent/agent/screen-state/{project_id}/{screen_id} for IR + backend/angular metadata consistency.
6) On error:
   - If 400 unstable/failed: show message + disable that version option in the UI (if listed).
   - If 404 version missing: show message and prompt user to pick another version.

API layer (frontend helpers)
---------------------------
